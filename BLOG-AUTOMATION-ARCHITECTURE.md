# Architecture: Folder Structure, Tech Stack, Naming Conventions

## Folder Structure

```
blog-pipeline/
├── prisma/
│   ├── schema.prisma                 # data model + datasource (postgres)
│   └── migrations/                   # generated by `prisma migrate dev`
├── .github/
│   └── workflows/
│       └── publish-cron.yml          # GH Actions cron trigger
├── src/
│   ├── research/
│   │   ├── searchConsole.ts          # GSC API client + impression/query pulls
│   │   ├── trends.ts                 # google-trends-api wrapper
│   │   └── topicScorer.ts            # ranks/filters candidate topics
│   ├── competitor/
│   │   ├── serpFetch.ts              # gets top-ranking URLs for a keyword
│   │   ├── pageScraper.ts            # cheerio/axios static scrape
│   │   ├── jsRenderScraper.ts        # playwright fallback (JS-rendered pages only)
│   │   └── gapAnalyzer.ts            # diffs structure, flags missing/common subtopics
│   ├── generate/
│   │   ├── deepseekClient.ts         # OpenAI-compatible client pointed at DeepSeek
│   │   ├── outlineGenerator.ts       # outline prompt + structured JSON parse
│   │   └── draftGenerator.ts         # section-by-section draft generation
│   ├── review/
│   │   ├── exportToMarkdown.ts       # writes draft to local .md for review
│   │   ├── exportToGoogleDocs.ts     # optional: Google Docs API export
│   │   └── reviewStatus.ts           # status flag read/write helpers
│   ├── seo/
│   │   ├── metaFormatter.ts          # title/meta description/slug
│   │   ├── internalLinker.ts         # picks + varies internal link anchors
│   │   └── schemaMarkup.ts           # JSON-LD Article/BlogPosting builder
│   ├── publish/
│   │   ├── shopifyPublisher.ts       # Shopify Admin API article push
│   │   └── cmsPublisher.ts           # alt CMS adapter (if not Shopify-hosted)
│   ├── db/
│   │   ├── client.ts                 # PrismaClient singleton
│   │   ├── topicsRepo.ts             # CRUD for Topic model
│   │   ├── draftsRepo.ts             # CRUD for Draft model
│   │   └── publishedRepo.ts          # CRUD for PublishedPost model
│   ├── tracking/
│   │   └── performanceSync.ts        # weekly GSC pull → updates published_posts
│   └── shared/
│       ├── types.ts                  # shared interfaces (Topic, GapReport, Draft, etc.)
│       ├── config.ts                 # env var loading, constants
│       └── logger.ts                 # simple structured logging
├── scripts/
│   ├── run-research.ts               # CLI entry: Stage 1
│   ├── run-gap-analysis.ts           # CLI entry: Stage 2
│   ├── run-generate.ts               # CLI entry: Stage 3+4
│   ├── run-publish.ts                # CLI entry: Stage 6+7 (only on approved drafts)
│   └── run-tracking.ts               # CLI entry: Stage 8
├── drafts/                           # exported .md files for human review (gitignored)
├── .env.example
├── .gitignore
├── package.json
├── tsconfig.json
└── README.md
```

**Why this shape**: each pipeline stage from strategy.md maps to one top-level folder under `src/`. `scripts/` holds thin CLI entry points that import from `src/` and get called by cron — keeps orchestration separate from logic, so each piece is independently testable.

## Tech Stack

| Layer                | Tool                                                                                        | Notes                                            |
| -------------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| Language             | TypeScript                                                                                  | matches existing skillset                        |
| Runtime              | Node.js (LTS)                                                                               |                                                  |
| Scheduler            | GitHub Actions cron                                                                         | free, no server to maintain                      |
| Keyword/trend data   | `google-trends-api`, Google Keyword Planner (manual export)                                 | free                                             |
| Search performance   | `googleapis` (Search Console API)                                                           | free, requires GSC property verified             |
| Static scraping      | `axios` (or `undici`) + `cheerio`                                                           | default scraping path                            |
| JS-rendered scraping | `playwright`                                                                                | fallback only, confirm via view-source first     |
| LLM                  | DeepSeek-V4-Pro, OpenAI-compatible SDK (`openai` npm) pointed at `https://api.deepseek.com` |                                                  |
| Database             | Prisma + Postgres (Neon/Supabase/Vercel Postgres)                                           | site is on Vercel, no persistent disk for SQLite |
| CMS publish          | `@shopify/shopify-api` (if Shopify-hosted)                                                  | swap adapter if blog lives elsewhere             |
| Doc export (review)  | `googleapis` (Docs API) or plain `fs` for markdown                                          | markdown is simpler, start there                 |
| Env management       | `dotenv`                                                                                    |                                                  |
| Validation           | `zod`                                                                                       | validate LLM JSON output before DB insert        |
| Testing              | `vitest`                                                                                    | lightweight, TS-native                           |
| Linting/formatting   | `eslint` + `prettier`                                                                       |                                                  |

No Redis, no message queue, no separate backend service — deliberately excluded, see strategy.md. SQLite dropped in favor of Prisma+Postgres since the app is deployed on Vercel (no persistent filesystem).

**Execution split**: pipeline scripts (`scripts/run-*.ts`) still run via GitHub Actions cron, not as Vercel functions — scraping + LLM generation can exceed Vercel's execution time limits on lower tiers. GH Actions writes to Postgres via `DATABASE_URL`; the Vercel-hosted site only reads published posts from the same DB (or Shopify Admin API).

## Naming Conventions

**Files**: `camelCase.ts` for modules (`gapAnalyzer.ts`), `kebab-case.ts` for CLI scripts (`run-gap-analysis.ts`) — scripts are entry points, treat them like commands.

**Functions/variables**: `camelCase`. Functions are verb-first: `fetchTopRankingUrls`, `scoreTopicPriority`, `exportDraftToMarkdown`. Avoid vague names like `process()` or `handle()`.

**Types/interfaces**: `PascalCase`, no `I` prefix (`Topic`, not `ITopic`). Match table names singularized: table `topics` → type `Topic`.

**DB models** (Prisma schema): `PascalCase`, singular (`Topic`, `GapReport`, `Draft`, `PublishedPost`). Prisma maps these to plural snake_case tables via `@@map("topics")` etc. — set this explicitly in schema.prisma rather than relying on default pluralization.

**DB fields**: `camelCase` in schema.prisma (Prisma convention), mapped to `snake_case` columns via `@map("search_console_impressions")`. Keeps TS-side code idiomatic while Postgres columns stay snake_case.

**Env vars**: `SCREAMING_SNAKE_CASE`, prefixed by service (`DEEPSEEK_API_KEY`, `SHOPIFY_ADMIN_TOKEN`, `GSC_CLIENT_EMAIL`).

**Status enums** (used across `drafts`, `published_posts`): lowercase snake strings — `pending`, `pending_review`, `approved`, `rejected`, `published`. Keep this exact set; don't let ad-hoc statuses creep in across modules.

**Branches/commits** (if using git flow): `feat/gap-analyzer`, `fix/shopify-publish-retry` — standard conventional-commit style, since you're already used to this from app dev.
